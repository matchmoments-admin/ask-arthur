# Shop Guard plan — critique

> Critique of `docs/plans/shop-guard.md` (read at 2026-05-18). Line references are to that file as it stood at the time of this review. Do not edit the plan in place — record decisions back into the plan via a follow-up PR.

## Executive summary

The plan is unusually disciplined for a v0: it correctly chooses the Charity Check Result shape over a free-form `enrichment` blob (§2), correctly puts the new write-hot table outside `scam_reports` (§4), correctly puts embeddings on a sibling per ADR-0005 (§4), and is honest about the three-implementations-of-a-pillar-engine debt (§8.7). The PR sequence is dependency-clean and individually <500 LOC. However it has a handful of real problems that warrant rework before PR 1: it invents a `verdict_enum` Postgres type that does not exist in this codebase (§4 SQL); it conflates Stage-3 `<all_urls>` host permission with the plan's main rollout while only making it a "stretch PR 9", which understates the CWS-rejection risk; the cost-brake of A$5 is implausibly low against the same plan's own Netcraft cost estimate; and the plan is silent on bot integration and B2B API exposure — two surfaces that the system map already calls out as load-bearing for SPF-regulated buyers. There are also two cases of banned vocabulary ("factor", "indicator" never appear but "case" leaks in twice and a `factor` candidate sneaks in through pillar weights). Overall **APPROVE WITH CHANGES** — the structural decisions are right; the SQL detail, the rollout staging realism, and the missing surfaces need fixing before PR 1 starts.

---

## Findings

### CRITICAL

1. **CRITICAL — §4 SQL (line 116): `verdict_enum` does not exist in this codebase.** `scam_reports.verdict` is `TEXT NOT NULL CHECK (verdict IN ('SAFE','SUSPICIOUS','HIGH_RISK'))` per migration v21 line 14; no Postgres `CREATE TYPE … AS ENUM` for verdicts has ever shipped. The plan's `verdict verdict_enum NOT NULL` will fail at `apply_migration` time. **Fix:** either (a) match the existing `CHECK (verdict IN (...))` shape exactly, including `UNCERTAIN` if Shop Guard wants the four-level verdict (which the plan claims in §2 line 19 — note that `scam_reports` currently does NOT have `UNCERTAIN` in its check constraint, so this needs reconciliation), or (b) introduce a real `CREATE TYPE verdict AS ENUM (...)` in PR 2 and migrate `scam_reports.verdict` to it in a follow-up — but that's a non-trivial cross-table change and should not be hidden inside a "Shop Guard PR 2".

2. **CRITICAL — §2 line 19 + §4 line 116: four-level Verdict mismatch with `scam_reports`.** Plan declares Shop Check Result emits `SAFE | UNCERTAIN | SUSPICIOUS | HIGH_RISK` (matches CONTEXT.md canonical 4-level Verdict, good). But `scam_reports.verdict` only allows 3 values today (no `UNCERTAIN`). §8.8 (line 280) proposes linking Shop Check Result to a Scam Report — but the FK target can't carry an `UNCERTAIN` Verdict. The plan does not flag this. **Fix:** either widen the `scam_reports.verdict` check constraint in a separate migration before PR 8, or document that Shop Check Result → Scam Report linkage only fires when the Shop Check verdict is one of the three legacy values. The current Charity Check has the same constraint mismatch (it emits 4 levels, never links back), and that is the design the plan should match explicitly, not paper over.

3. **CRITICAL — §3 Extension lines 80–86 + §6 line 235 (Stretch PR 9): `<all_urls>` is being marketed as "stretch" but is necessary for Stage-3 detection rate target.** §7 line 247 promises ≥85% detection at Stage 3, "+ extension stage-3 overlay". Stage-3 requires the manifest v1.0.2 bump with `<all_urls>` host_permissions — i.e. the Chrome Web Store sensitive-permission re-review (1–3 days, re-rejection risk per CLAUDE.md deployment table). Hiding this in "Optional PR 9" makes the rollout plan optimistic in a way the research brief explicitly warns against. **Fix:** promote PR 9 into the main sequence and gate Stage-3 success metric on it; or honestly state that without Stage-3 the plan tops out at the Stage-2 ≥80% target. Also note: the Chrome Web Store re-rejection failure mode (and what we ship if it happens) is missing from §8 Risks beyond a one-line mention in §8.4.

### IMPORTANT

4. **IMPORTANT — §4 line 149 + §8.2 line 268: cost brake is inconsistent with the plan's own Netcraft estimate.** §8.2 estimates Netcraft median $0.20/lookup; A$5/day cap → ~25 lookups/day before pause. Stage-2 (§7 line 246) is "10% rollout" — at site-wide consumer traffic of even a few thousand commerce-URL analyses a day, 10% = hundreds of lookups, blowing the cap inside an hour. The brake will fire constantly and Shop Guard will spend its life paused. Compare with `REDDIT_INTEL_CAP_USD=10` (a low-volume cron) and `PHONE_FOOTPRINT_CAP_USD=5` (rate-limited B2B). Shop Guard is a high-volume consumer enrichment, the comparison group is wrong. **Fix:** before PR 1, get a Netcraft pricing call done (it's listed as an "open question" but the cap is being set without the data); pick a cap that matches realistic Stage-2 volume × per-call cost + headroom; document the cap derivation in `docs/ops/shop-guard-config.md` (which the plan does not yet propose creating — see finding 13).

5. **IMPORTANT — §3 line 92: bot integration is entirely absent.** The system map (`docs/system-map/web-surface.md` lines 285–294) shows five bot webhooks (Telegram, WhatsApp, Slack × 2, Messenger). Paste-the-URL into a bot is one of the canonical scam-detection moments for shoppers, and `@askarthur/bot-core/analyze` already routes through `/api/analyze`. The plan claims `/api/analyze` is the single entry point (§4 line 99) — which means Shop Guard does work for bots **by accident**, but the plan does not (a) confirm the bot path renders the new Pillar block in any of the four platform-specific formatters, (b) confirm the bot rate-limit/cost-telemetry tagging is correct, or (c) decide whether bots show the full Pillar accordion or a one-line summary. **Fix:** add a PR (or fold into PR 6) for bot-formatter Shop Check rendering. At minimum, write a test that asserts `toTelegramMessage` and `toSlackBlocks` don't crash when the analyze result carries a `shop_check` block.

6. **IMPORTANT — §3 line 92 + §6 PR list: no B2B `/api/v1/...` exposure.** The Reddit Intel plan exposes `/api/v1/intel/*`; Phone Footprint exposes B2B endpoints; Charity Check has `/api/charity-check`. The research brief specifically calls out SPF-regulated entities (banks, telcos) as a strategic surface — these buyers want a programmatic check for cards-on-file merchant URLs at transaction time. The plan never mentions `/api/v1/shop-check`. **Fix:** add a stretch item (Optional PR 11) for B2B exposure behind `NEXT_PUBLIC_FF_SHOP_GUARD_B2B_API` (matching the `scamsSearchB2bApi` / `redditIntelB2bApi` flag pattern), gated by API-key tier. Even if not shipped in v0.1, it should be in the plan as an explicit "not in scope" decision with the reason.

7. **IMPORTANT — §3 line 92 + §8 Risks: no two-way intel sharing with NASC / Scamwatch.** Scamwatch already flows in via `pipeline/scrapers/scamwatch_alerts.py` (consumed in §7 line 255 for the adversarial corpus), but there is no outbound path. When Shop Guard establishes HIGH_RISK on a URL that's not yet in Scamwatch's feed, that's net-new intelligence Ask Arthur could share. The research brief flags this as strategically valuable. **Fix:** add a §8.11 open question for the onward-reporting path (the `onward_report_log` table already exists per database.md line 99); a one-liner deciding "not in v0.1, but the schema supports it via `source_surface='shop-guard'`" is enough.

8. **IMPORTANT — §3 line 56–60 vs §3 line 90–95: the asymmetry with Charity Check is asserted but the justification is thin.** Charity Check is a standalone `/charity-check` page with autocomplete because it answers a "named-entity claim" question (line 57). Shop Guard is in-drawer because it answers a "reactive moment of doubt" (line 58). Plausible — but Charity Check ALSO has a v0.2e "main-checker hook" (per CLAUDE.md Quick Reference) that means Charity Check ALSO runs from `/api/analyze` for the reactive moment. So the asymmetry isn't standalone-vs-in-drawer; it's "does Shop Guard need a discoverable, bookmarkable, autocomplete-driven landing page?" Plan says no. **The reactive case is the dominant one — accept that, but verify against analytics on Charity Check's traffic split.** **Fix:** before PR 6, pull `/charity-check` page views vs `/api/charity-check` calls from analyze flow; if the page is <10% of charity-check usage, the asymmetry is justified; if it's >30%, ship a `/shop-check` page in v0.2.

9. **IMPORTANT — §4 line 153 + §7 line 247 + research-brief constraint: the cheap-pillar 5s inline budget will not hold under Stage-3 traffic.** Plan claims cheap pillars run inline in 5s (line 153). Cheap pillar list is `safe_browsing` + `domain_age` + `historical_reports` + `abr_au`. Each is a network round-trip: Safe Browsing is bundled in `/api/analyze` step 7 already (sub-second), domain-age WHOIS/RDAP can take 2–8s with public RDAP endpoints, ABR can be 1–3s, `historical_reports` is a `scam_entities` JOIN (sub-second). Worst case is dominated by RDAP. **Fix:** either move `domain_age` to the heavy/Inngest path or commit to a paid RDAP provider (`RDAP_PROVIDER_KEY`) and measure p50/p95 in PR 3 unit tests with the real provider, not just stubs. Also: research brief's <300ms p50 content-script overhead target needs explicit measurement in PR 7 (the plan says "tested under stage-1 perms" but doesn't say "latency measured" — this should be in PR 7 Exit criteria).

10. **IMPORTANT — §4 line 137 + ADR-0005 wording: plan correctly defers HNSW to a sibling, but should explicitly forbid GIN trigram on `shop_checks.url_normalized` until row count is known.** `acnc_charities` GIN trigram on legal-name works because it's read-heavy with weekly writes; `shop_checks` is write-frequent (every commerce URL analyze sees) and any GIN there will dirty index pages on every insert. The plan mentions HNSW exclusion but not GIN. **Fix:** add to §4 line 137: "No GIN trigram on `url_normalized` either; if fuzzy URL lookup is needed, do it on the sibling embeddings table or via a materialised view refreshed nightly."

11. **IMPORTANT — §2 line 12 + §6 PR 3: `factor` / pillar-weight language in §2 table (lines 35–44).** The table header says "Weight (initial)" which is fine, but the body text on line 46 says "Pillar scores are 0..100, higher = more risk (matches Phone Footprint and Charity Check)". That's good. However the §8.7 open question (line 278) says "the generic shape" — fine. The plan is mostly clean. The one slip: §2 line 18 talks about "the per-Pillar `{ id, score, confidence, available, reason?, detail? }`" — `score` and `confidence` are correct CONTEXT.md vocabulary (composite Pillar shape from ADR-0002). No banned vocabulary actually appears. **Withdrawn — this is a nit only:** the §6 PR 3 description (line 206) says "factor the AU-specific call into a shared helper if and only if two callers actually need it" — `factor` here is the English verb, not a domain noun. Acceptable.

12. **IMPORTANT — §4 line 121: `source_surface text` should be a constrained text or enum.** Free-form text invites typo-driven data fragmentation ("mobile-share" vs "mobile_share" vs "ios-share"). The values are enumerable: `web | extension | mobile-share | bot-telegram | bot-whatsapp | bot-slack | bot-messenger`. **Fix:** add `CHECK (source_surface IN (...))` matching the existing `feed_items.source` pattern from `docs/system-map/database.md` line 34.

13. **IMPORTANT — no `docs/ops/shop-guard-config.md` proposed.** Phone Footprint, Charity Check, Voyage embeddings, Reddit Intel all have ops-config docs (per CLAUDE.md Quick Reference). The plan adds env vars, feature flags, cost brakes, and a Stripe-adjacent decision (no Stripe SKUs in v0.1, presumably) without an ops-config landing place. **Fix:** PR 1 should create a stub `docs/ops/shop-guard-config.md` with the env-var table and a "flip-on checklist" mirroring the Phone Footprint config doc.

14. **IMPORTANT — §4 line 152 ("Inngest fan-out … `FF_ANALYZE_INNGEST_WEB` pattern"): does this depend on Inngest being on for analyze?** `FF_ANALYZE_INNGEST_WEB` is currently OFF (default) per `apps/web` env. If Shop Guard's heavy pillars rely on the analyze Inngest path being live, the dependency must be documented. If Shop Guard ships its own Inngest events independently, that's fine but the plan should say so explicitly. **Fix:** add a sentence to §4 line 152 clarifying: "Shop Guard emits its own `shop.check.completed.v1` event regardless of `FF_ANALYZE_INNGEST_WEB` state; only the analyze-side persistence hooks depend on that flag."

### NIT

15. **NIT — §1 line 5: "Shop Check Result" is correctly introduced; §2 line 31 instructs to add it to CONTEXT.md.** Good. Note: §6 PR 1 (line 196) says "Add `Shop Check Result` to CONTEXT.md" — confirm this lands in PR 1, not later. The Reddit Intel plan added vocabulary in a late PR and the codebase carried inconsistent language for a fortnight; PR 1 is the right time.

16. **NIT — §2 line 13: "two adapters mean a real seam" is mis-attributed to ADR-0002.** That phrase is in `.claude/skills/improve-codebase-architecture/LANGUAGE.md`, not in the ADR itself. ADR-0002 says "two implementations share roughly 80%" and chooses NOT to extract. Both are right but the citation is off. **Fix:** cite LANGUAGE.md for the seam-extraction principle and ADR-0002 for the "deferred extraction" decision. The §6 line 206 reference is correct ("the third pillar-typed engine; flag in PR description that the shared-module case is now ripe"); align §2 with that framing.

17. **NIT — §3 line 64 ("the server branches into the Shop Guard fan-out *in parallel* with the existing Claude + URL reputation phases"): confirm this is `Promise.allSettled` not `Promise.all`.** A failing Shop Guard branch must not crater the analyze response. Plan doesn't say. **Fix:** §4 PR 4 exit criterion should add "uses Promise.allSettled at the route level; Shop Guard rejection logs but does not propagate to the user-visible response".

18. **NIT — §6 PR 7 (lines 225–228): missing kill-switch test.** The plan has three flags (`NEXT_PUBLIC_FF_SHOP_GUARD`, `FF_SHOP_GUARD_BACKEND`, `WXT_SHOP_GUARD`) and a feature_brakes row. The kill-switch story (turn the feature off without a deploy) is: flip `FF_SHOP_GUARD_BACKEND` to false in Vercel env, or `INSERT INTO feature_brakes(feature, paused_until) VALUES ('shop_guard', now()+interval '7 days')`. Plan doesn't document the latter. **Fix:** §5 line 189 add a one-liner: "Emergency kill-switch without deploy: insert a `feature_brakes.shop_guard` row with a far-future `paused_until`."

19. **NIT — §6 PR 5 line 217: "per-function header comments document expected duration to satisfy pg-watchdog runbook" is exactly the right shape.** Echo this in PR 4 too — the cheap-path orchestrator can also slow-burn if RDAP wedges. CLAUDE.md "Always Do" calls for any new Inngest function or cron route that could exceed 10 min to be documented; the rule applies to the route handler too.

20. **NIT — §7 line 253–255: the AU SMB corpus sources are mostly third-party.** "Manually-vetted Etsy AU sellers" + "Shopify made-in-Australia curated list" are vendor-curated, not authoritative. Mixed with the ABR + ASIC sample (authoritative), the FP-cap of 2% on a curated mixed corpus could be hiding a higher FP rate on the long tail. **Fix:** add to §7 a "blind-spot" note: "the 2% FP cap is measured on the curated corpus; before Stage-2 flag-on, sample 100 random Australian e-commerce URLs from real `/api/analyze` traffic (filter `source_surface='web'` + url-detector classified as `url`) and verify the long-tail FP rate is also ≤2%."

21. **NIT — §8 Risk 10 (line 284): "PR 6 explicitly excludes the audit grade from the Shop Guard card" — good.** Worth strengthening: the `/site-audit` page (apps/web `/api/site-audit`) should also NOT cross-link to Shop Guard in either direction. They're orthogonal questions per CONTEXT.md (Unified Scan Result vs Verdict). Add a one-liner to PR 6 that audits don't render a Shop Guard pill and Shop Guard cards don't render an audit pill, even when both are available.

22. **NIT — §4 line 156 ("may itself enqueue follow-up entity enrichment"): unbounded recursion risk.** `shop-guard-infrastructure-cluster` enqueuing entity enrichment is fine; entity enrichment is already idempotent (per `packages/scam-engine/src/inngest/entity-enrichment.ts`). But the plan should explicitly cap the depth — entity enrichment must not re-enqueue Shop Guard. **Fix:** add to PR 5 description that the entity-enrichment events Shop Guard emits carry `{ source: 'shop-guard', depth: 1 }` so a downstream handler can refuse to re-enter Shop Guard.

23. **NIT — §6 PR 3 (line 206): "Reuses ABR provider shape from charity-check (factor the AU-specific call into a shared helper if and only if two callers actually need it — per the deletion test, inline first)".** Good — deletion test applied. Worth noting in §8.7 that ABR adapter sharing is also a candidate for the eventual extraction PR, not just the orchestrator/scorer.

24. **NIT — §3 line 73 (`commerce-detector.ts` two-of-three rule): confirm that pages with payment forms but NO Product schema are still detected.** Many Australian small-shop checkout pages skip schema.org Product on the cart/checkout step. The two-of-three rule says any 2 of {schema.org Product, payment form, platform DOM marker} — a checkout step has the payment form + the platform marker, so it'd fire. But a non-Shopify/Woo/BigCommerce store with hand-rolled cart + schema.org Product on the product page but no platform marker only has 1 of 3 on the product page. **Fix:** add unit-test cases in PR 7 for hand-rolled AU artisan shops (likely sourced from Etsy AU sellers in the SMB corpus).

25. **NIT — §6 PR 2 line 200: RLS policy spec.** "service-role write, public read of own request_id only — pattern from `phone_footprints`". Confirm: `phone_footprints` RLS allows authenticated users to read their own rows by user_id, not anonymous read by request_id. Shop Guard is largely anonymous, so the RLS shape is closer to `verified_scams` (service-role write, authenticated read-all via API key tier) than `phone_footprints`. **Fix:** clarify in PR 2 which existing table's RLS pattern actually fits — and write it down so the reviewer can copy-paste.

---

## Verdict on the plan

**APPROVE WITH CHANGES.**

Top 3 issues to fix before PR 1:

1. **Replace `verdict_enum` with the existing `TEXT + CHECK` shape** (finding 1) and reconcile the four-level Verdict with the three-level `scam_reports.verdict` constraint (finding 2). This is a SQL-correctness blocker — the migration as written will not apply.
2. **Cost brake A$5 is wrong for the projected volume given the Netcraft median.** Get the Netcraft pricing call done and resize the cap before PR 5 starts (finding 4). The cap derivation belongs in a new `docs/ops/shop-guard-config.md` (finding 13).
3. **Decide on bot integration and B2B API exposure now, not later** (findings 5 and 6). Even if both are explicitly out of scope for v0.1, the plan needs to record that decision with the reason — `/api/analyze` is the entry point for all five bot platforms today, and Shop Guard will silently run for them whether the plan addresses it or not.

The structural decisions (Charity-Check shape, sibling-table for embeddings, deletion-test against a free-standing route, in-drawer surface, staged extension permissions, three-layer idempotency, chunked retention) are all correct and reflect the codebase's hard-won rules. After the three fixes above and a sweep of the nits, the plan is ready to start shipping.
