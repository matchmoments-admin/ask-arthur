# packages/types ŌĆö local guide

Scoped guidance for the `@askarthur/types` package ŌĆö Zod 4 schemas and TypeScript interfaces that form the cross-package contract surface. Read this in addition to the [root CLAUDE.md](../../CLAUDE.md).

## What this package owns

- **Domain Zod schemas** ŌĆö `analysis.ts`, `scanner.ts`, `intelligence.ts`, `news-intel.ts`, `extension.ts`, `extension-security.ts`, `site-audit.ts`.
- **Tier / billing types** ŌĆö `billing.ts` (incl. `TIER_LIMITS` and `TierLimit` shapes consumed by `apps/web` API routes).
- **Organisation types** ŌĆö `organization.ts` (multi-tenant org / member shapes).
- **Generated DB types** ŌĆö `db.generated.ts` (generated by `mcp__supabase__generate_typescript_types`; check in but don't hand-edit).
- **Index** ŌĆö `index.ts` re-exports everything.

## What it doesn't own

- **Validation _calls_.** This package exports schemas; the call sites that validate live in `apps/web/app/api/**` route handlers, Inngest functions, and bot route guards. This package provides the schema, not the parse.
- **Database access.** Zod schemas describe the wire shape; reading/writing rows lives in `apps/web` and `packages/scam-engine`.
- **Bot formatting / Claude analysis / RPC bodies.** Those packages depend on this one ŌĆö never the other way around.

## Critical rule: schemas are the cross-package contract

**Any Inngest event payload, API route input, or Supabase RPC input that crosses a package boundary must validate against a schema exported from `@askarthur/types`.**

Inline `z.object({...})` defined outside this package and used across packages is a smell ŌĆö extract it here so the producer and consumer agree.

If you find yourself writing `z.object` in a route handler, ask: does another package read this shape? If yes, the schema belongs in `@askarthur/types`. If no (the shape never leaves the file), inline is fine.

## Public API surface (representative)

The package is one big re-export ŌĆö `import { ... } from "@askarthur/types"` for anything; sub-path imports (`@askarthur/types/billing`, `@askarthur/types/scanner`) are also available for tree-shaking.

| Export                  | Purpose                                                   | Defined in              |
| ----------------------- | --------------------------------------------------------- | ----------------------- |
| `AnalysisResult`        | Canonical verdict shape returned by `/api/analyze`        | `analysis.ts`           |
| `AnalysisRequest`       | Input payload shape                                       | `analysis.ts`           |
| `UnifiedScanResult`     | Output of the scanner family (extension, MCP, site-audit) | `scanner.ts`            |
| `TIER_LIMITS`           | Per-tier API quotas (free / starter / growth / scale)     | `billing.ts`            |
| `ExtensionSecurityScan` | `@askarthur/extension-audit` payload shape                | `extension-security.ts` |
| `NewsIntelTheme`        | Theme cluster shape from the Reddit/news intel pipeline   | `news-intel.ts`         |
| `Database`              | Supabase-generated DB types                               | `db.generated.ts`       |

## Scoped commands

```bash
pnpm --filter @askarthur/types typecheck
```

No test suite ŌĆö schemas are exercised by their consumers' tests.

## Gotchas

- **Zod 4 removed `z.string().url()`.** Use `z.url()` (top-level). Same for `z.email()`. Reference: [`grill-with-docs/STACK-PINS.md`](../../.claude/skills/grill-with-docs/STACK-PINS.md).
- **`db.generated.ts` is regenerated, not hand-edited.** Regenerate with `mcp__supabase__generate_typescript_types` after migrations. Hand edits get clobbered.
- **Sub-path imports beat barrel re-imports** for tree-shaking ŌĆö `import type { TierLimit } from "@askarthur/types/billing"` is preferred over `from "@askarthur/types"` in apps that care about bundle size (extension, mobile).
